# ADR: OpenCode plugin-native distribution (retire install.sh)

## Status
Accepted; slash-UX mechanism superseded by 20260611-opencode-slash-commands-use-plugin-wrappers

## Context

The OpenCode target ships a curl/`install.sh` installer that copies skills and generated slash commands into `~/.config/opencode/` with install-time suffix namespacing (`-manifest-dev` / `-manifest-dev-tools`, driven by `component-namespaces.json` + `install_helpers.py`). Three developments broke that design:

1. **Native skills-as-commands.** Since OpenCode v1.1.48 (Feb 2026), every discovered skill is auto-exposed as a slash command (anomalyco/opencode#12109). The installer's generated `commands/` now produce *duplicate* menu entries (`/define-manifest-dev` the command + `/define-manifest-dev` the native skill entry) — the installer became actively worse, not just redundant.
2. **Cross-CLI bleed through shared Agent Skills directories.** OpenCode scans `.claude/skills/` and `.agents/skills/` (project and home). Anything manifest-dev installs there is also visible to Pi (itself an Agent Skills host). The conflict is structural, not cosmetic: per-host skill sets genuinely differ — Pi deliberately ships **no** `do`/`done`/`escalate`/`auto`/`babysit-pr` skills (runtime-owned via Harness-level Do), while OpenCode needs them as skills. A shared directory cannot express "exists for OpenCode, must not exist for Pi." Per-host private channels are required.
3. **Plugins can now carry skills.** OpenCode plugins are JS/TS hook modules (not asset bundles like Codex plugins — no marketplace, no git install). But at HEAD, config supports `skills.paths` (custom skill directories), and a plugin's `config` hook mutates config — so a plugin can register skills bundled inside its own package, fully contained, invisible to other CLIs. (Docs lag source: docs/skills still doesn't mention `skills.paths`; it's read in `packages/opencode/src/skill/index.ts`.)

Source-verified constraints that shaped the install channel: OpenCode plugin specs accept **npm names and local file paths only** — no git URLs (`packages/opencode/src/plugin/shared.ts`), unlike Pi's `pi install git:`. The config-only `skills.urls` mechanism caches on first pull with no TTL/refresh, freezing users on day-one skills. The maintainer rejects an npm publish as new operational surface (account, tokens, release flow — the Pi package is git-installed, never published).

## Decision

Distribute manifest-dev to OpenCode as an **OpenCode plugin**, generated by `/sync-tools` under `dist/opencode/`: the OpenCode-flavored skills plus a small plugin entry (package.json + index) whose `config` hook appends the package-local `skills/` directory to `skills.paths` (and registers the AGENTS.md context via `instructions`). **Retire `install.sh`, `install_helpers.py`, `component-namespaces.json`, and the generated `commands/`** — native skills-as-commands replaces command generation, and the plugin boundary replaces suffix namespacing.

Install is **repo-clone + file path** (no npm publish): users clone the repo and add the plugin's absolute path to the `plugin` array in their global `opencode.json`; updates are `git pull`. `sync-tools` reclassifies OpenCode from "installer target" to "plugin-native target."

Result across all four hosts — Claude Code plugin, Codex plugin marketplace, Pi git package, OpenCode plugin — manifest-dev never installs into shared Agent Skills directories, and Pi needs no config changes.

## Alternatives Considered

- **Keep the installer**: degraded post-v1.1.48 (duplicate slash entries), pollutes the user's global config dir, and keeps namespacing machinery alive — rejected.
- **Skills-native via `npx skills add`** (tree URL into `~/.config/opencode/skills/`, the initially recommended route): no publish and repo-direct, but writes 18 skill dirs into user-global visible space, has no managed update/uninstall, and the project-scope default (`.agents/skills/`) recreates the Pi bleed — rejected after the cross-CLI conflict surfaced.
- **Config-only `skills.urls`** pointing at raw.githubusercontent: zero install footprint, but cache-miss-only pull freezes skills at first fetch — rejected.
- **npm publish of the plugin** (`"plugin": ["@doodledood/manifest-dev-opencode"]`): best end-user UX (one line, semver updates), rejected *for now* as unwanted operational surface; purely additive later — the same generated package can be published unchanged if demand appears.
- **Plugin disk-sync variant** (plugin writes bundled skills into `~/.config/opencode/skills/` at startup): an installer in disguise; reserved as the fallback inside the same architecture if the `config`-hook ordering risk below materializes.

## Consequences

### Positive
- Zero shared-directory footprint: no Pi/OpenCode skill collisions, no `.agents` pollution, nothing for other CLIs to misdiscover.
- Deletes installer infrastructure (`install.sh`, `install_helpers.py`, `component-namespaces.json`, `commands/` generation); `/sync-tools` simplifies substantially.
- Uninstall is one config line + deleting the clone; the skill payload lives package-local, not interleaved with user-owned skills.
- All four host distributions converge on the same shape: native extension unit per host.

### Negative
- Users manage a clone and update via `git pull` — no managed install/update flow like `pi install`/`pi update` (OpenCode plugins simply don't support git sources).
- Skills carry bare names (`define`, `do`, `auto`); collisions with user-owned skills rely on OpenCode's first-found-wins dedup (warning logged, no namespacing mechanism exists).
- OpenCode honors only `name`/`description` frontmatter at HEAD — `user-invocable`/`disable-model-invocation` are ignored, so helper skills (`done`, `escalate`, `poll-slack`) appear in the slash menu and model context.
- **Verification gate before retiring the installer**: prototype that the plugin `config` hook runs before skill discovery reads `skills.paths` (ordering not conclusively established from source). If it fails, fall back to the plugin disk-sync variant — same architecture, same install UX.
- Minimum OpenCode version must be pinned (≥ v1.1.48 for skills-as-commands; the release introducing `skills.paths` must be identified at implementation).
- Existing installer users need a migration note (remove suffixed copies, or the old uninstall path) to avoid lingering duplicates.
