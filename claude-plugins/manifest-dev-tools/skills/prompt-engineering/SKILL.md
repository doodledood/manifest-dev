**User request**: $ARGUMENTS

Prompts are manifests: state the goal and the load-bearing constraints; trust the model to do everything else. Every word **steers** (behavior the model wouldn't reach on its own) or **scaffolds** (preempts a failure the clean posture handles); cut scaffold. The default is too few words — add specific lines back only when an observed failure justifies it. Aspirational shape: **1-3 short paragraphs of meat** in SKILL.md; contracts (schemas, formats, tables, workflow steps) pushed to `references/*.md`. The model already knows how to search, analyze, generate, format — don't restate. Decision rules over absolutes for judgment calls (reserve MUST/NEVER for true invariants — safety, output contract, hard constraints). No arbitrary numbers; principles instead.

**Branch on the request.** *Creating:* probe essence first — what's the job in one sentence, what anti-defaults need explicit steering (the behaviors the model wouldn't reach), what contracts must be exact (schemas, output formats, tool-invocation shapes). Then draft 1-3 paragraphs containing only essence + anti-defaults; push contract data to references. Hold positions under pushback when evidence supports them; when you infer a constraint from context, confirm before encoding. *Updating / slimming:* read the prompt, find the scaffolding (capability instructions, restating model defaults, examples for behaviors the model already does, arbitrary numbers, "why this exists" framing, multi-paragraph explanations of single ideas, tables for non-tabular content), cut it; preserve every load-bearing steer with the cut justified. *Reviewing:* for each line ask *"does this change behavior the model wouldn't produce without it?"* — flag the no's, propose the cut. *Diagnosing a failing prompt:* load `references/metaprompting.md` and follow its diagnose → surgical-revision workflow (pre-flight check first; metaprompting is the wrong tool for missing goals, missing capabilities, or visible single-bug typos).

**Specializations.** Skills are directories (SKILL.md + companions for contract data per `references/skills.md`); the frontmatter `description` is a **trigger spec** — what + when + trigger terms users actually say — under 1024 characters, not a human summary. Agents run isolated — they don't inherit parent tools; declare every required tool in `tools:` frontmatter (audit: read what the agent does, identify every capability explicit and implicit, verify all are declared). When essence-only doesn't fit a complex system prompt (multi-stage workflows, agents with subdomains, instructional documents), reach for `references/canonical-template.md` (7-section template) and `references/system-prompt-patterns.md` (technique library for non-trivial sections — verification loops, retrieval budgets, output contracts, ambiguity handling).

Discover context before drafting: hidden domain terms, edge cases, integration points, success criteria, constraints — through targeted questions, recommended-answer-per-ask. Keep emotional tone low-arousal — no urgency framing, no excessive praise, no pressure language; the opening tone propagates into model responses.
